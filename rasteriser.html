Software Rasteriser

Introduction

This project is my exploration of the concepts of rasterisation that I had learnt about in my Introduction to Computer Graphics course unit. I decided to use C++ for this project as I wanted to get more familiar with the language, and I didn't want to use python again.

It is currently able to render 3D models in .o files with a perspective projection. The models can also be transformed.


Project Overview

Before I could start on any of the maths for rendering, I needed a way to draw pixels to the screen. I wanted a simple low level, but well documented solution, so I decided on SDL. It uses a texture which contains pixel data shown on the screen, this is updated with a texture buffer which I can write to.

Once that was completed, I needed to set up some Vector and Matrix classes, since they are very important in computer graphics. I also created a coordinate struct to just hold coordinates since they are needed a lot and this results in less overhead. Before rendering 3D triangles, I wanted to try rendering a 2D triangle first. Since 3D triangles are projected to 2 dimensions, this code can be reused when rendering 3D triangles.

In order to rasterise a triangle, we need to check each pixel to see if it lies within the triangle. To do this, I use the barycentric method. We take two of the sides of the triangle, and join them with the point to make a triangle and find its signed area using vectors. We then do this with the other two triangles created with the point to get three signed areas. If all 3 of these signed areas are not negative then the point is in the triangle.

Now I wanted to render 3D triangles, so I needed to define a way to project 3D triangles onto a 2D plane. To start with I decided to use an orthographic projection and simple take the 3D triangles x and y coordinates and discard the z coordinate.

In order to get a better sense of the 3D projection, I wanted to render an actual model. So I wrote a class to store models in, which is instantiated by reading object (.o) files. I used this to read a cube object file.

Now that the orthographic projection was working, I wanted to create a perspective projection so objects have a sense of distance. This involved a process called perspective division where you divide your x and y coordinates by the z coordinate, which causes further away objects to appear smaller.

With this working, I wanted to try and render a more complex model. So I downloaded the object file for Blender's Suzanne model and attempted to render it

While it did render, I noticed a few issues

  • The performance was not great
  • Some of the triangles at the back of the monkey were rendering in front of it
To solve the first issue, I made an optimisation using bounding volumes. Instead of checking every pixel against every triangle, I only check the pixels that lie within the smallest rectangle that encompasses the triangle. This is done by checking pixels between the minimum and maximum x and y coordinates of the triangle

The second issue, I solved using a depth buffer. Whenever we draw a pixel, we check if it's z value is less than what is currently stored in the z buffer, which is a buffer the same size as the framestore that stores the z values of the pixels plotted. In order to get z values for the pixels, I needed to interpolate them from the 3 vertices of the triangle. I did this using the Barycentric areas as weights for each vertice's z value.

Future Improvements

While I am happy with the progress I have made so far, there are still a lot of improvements that can be made

  • The CPU is used for rendering rather than the GPU which limits performance greatly. This is because I was more interested in learning and applying the rendering techniques than making a super performant rasteriser. While I don't have any plans to utilise the GPU, in the future, I may work to include more parallelism within the system
  • The model reader can only read .o files that are made up of triangles. In the future I will set up a system to create triangle fans out of larger polygons
  • Currently the triangles are rendered as random colours, but I would like to instead set the colour and use lighting and shading techniques
  • Models are rendered with small gaps between polygons
